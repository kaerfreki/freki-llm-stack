## Method

- Same prompts and scenarios as the [Ollama harness](RESULTS.md).
- Per model: container recreated with that Hugging Face id, cold load
  measured as wall-clock to `/health`, then 1 discarded warm-up and
  3 measured runs; tables show the **median**.
- TTFT is client-side wall-clock to the first streamed token on
  `/v1/completions` (includes prefill, no network hop off-box).
- Token rates are client-side from the final stream `usage` object:
  prompt tok/s ≈ prompt_tokens / TTFT; generation tok/s =
  completion_tokens / wall-time after the first token.
  `temperature=0`, `seed=42`; each run's prompt gets a unique prefix.
- VRAM peak is total GPU memory used (sampled every 0.5 s), including
  a small desktop baseline **and** the KV-cache pool vLLM pre-allocates
  up to `gpu-memory-utilization`. That is why peaks sit near the card
  limit even for small models — unlike Ollama, which grows VRAM with
  demand. There is no per-request load and no CPU offload split.
- Compose serves with `--enforce-eager` (no CUDA graphs) so multimodal
  Qwen3.5 AWQ fits a 16 GB card alongside a desktop session; single-stream
  rates are slightly lower than a fully graph-captured server.
- Rows omitted from the default matrix on this host (see model map):
  `qwen3.5:9b-q8_0` (FP8 multimodal OOM), `gemma3:12b` (tight),
  `qwen3.6:35b` (Ollama already CPU-offloads ~half).
